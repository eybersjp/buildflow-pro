## Data Flow Diagrams

### Primary User Request Flow

```
User Browser
  → Next.js Edge Middleware (auth check)
    → Next.js Server Component or API Route
      → Service Layer (validate → authorize → execute)
        → Supabase (RLS enforced)
          → PostgreSQL
        ← Data returned
      ← Typed response
    ← Rendered HTML or JSON
  ← User sees result
```

### Authentication Flow

```
User submits credentials
  → POST /api/auth/sign-in (or Supabase Auth directly)
    → Supabase Auth validates credentials
      → Returns JWT + refresh token
    → Session stored in httpOnly cookie (via @supabase/ssr)
  → Redirect to /dashboard
    → Middleware reads cookie
      → getUser() validates JWT server-side
        → If valid: allow request
        → If invalid: redirect to /login
```

### Multi-Tenant Data Isolation Flow

```
Authenticated Request
  → getUser() → session.tenantId extracted from JWT
    → Service function called with { tenantId, userId, userRole }
      → Supabase query includes .eq('tenant_id', tenantId)
        → RLS policy also checks auth.jwt() ->> 'tenant_id'
          → Double isolation: application-layer + database-layer
            → Data returned only for correct tenant
```

## Surprising Connections

> Non-obvious relationships between components. These are the connections that cause bugs when developers assume independence.

| Component A | Component B | Hidden Relationship | Risk If Ignored |
|---|---|---|---|
| `middleware.ts` | `getUser()` | Middleware must call `getUser()` (not `getSession()`) to validate JWT server-side | `getSession()` in middleware doesn't re-validate expired tokens |
| `audit_log` table | All service mutations | Every service function writes to `audit_log` — slow audit writes affect mutation speed | Audit log bottleneck affects perceived write performance |
| `RLS policies` | `service_role` client | RLS is bypassed when using the service role client — no tenant isolation | Using service role for user queries = tenant data leakage |
| `NEXT_PUBLIC_*` env vars | Build time | These are embedded at build time, not runtime — changing them requires a full rebuild | Changes to `NEXT_PUBLIC_*` in Vercel dashboard do not take effect until next deployment |

## Graph Update Protocol

When the architecture changes, update this file:

1. Update the **God Nodes** table if a new high-centrality component was added
2. Update the **Component Map** with new services, routes, or tables
3. Update **Surprising Connections** if a new non-obvious dependency was discovered
4. Update the **ADR Index** with the new decision
5. Set **Last Graph Update** to today's date
6. Write a new ADR in `docs/ADR/` documenting the change
7. Update `.antigravity/memory/decisions.md`

**Do not delete historical entries.** Mark superseded entries as `[SUPERSEDED by ADR-XXX]` instead.
